# Voice Wakeup Tester 使用文档

## 1. 工具说明

`Voice Wakeup Tester` 是一个用于智能眼镜语音唤醒率测试的 Windows 工具，支持：

- GUI 图形界面操作
- 命令行无界面运行
- RTOS 串口日志监听
- Qualcomm `adb logcat` 日志监听
- 人工嘴与噪声音箱独立输出
- 批量场景测试
- 结果报告自动导出

## 3. GUI 使用流程

### 3.1 选择平台

平台支持两种：

- `rtos`
- `qualcomm`

区别：

- `rtos` 使用串口监听日志
- `qualcomm` 使用 `adb logcat` 监听日志

### 3.2 选择音频输出设备

需要配置两个输出：

- `人工嘴`：播放唤醒词音频
- `噪声音箱`：播放噪声音频

建议：

- 人工嘴优先使用有线输出，降低蓝牙延迟抖动
- 噪声音箱可以使用蓝牙音箱
- 蓝牙设备优先选择 `Stereo/A2DP`，避免 `Hands-Free/AG Audio`
- 蓝牙设备重连后请点击 `刷新音频/蓝牙设备`

### 3.3 配置设备连接

如果平台为 `rtos`：

- 选择串口号
- 设置波特率

如果平台为 `qualcomm`：

- 选择 `ADB` 设备序列号

### 3.4 配置匹配规则

匹配规则用于判断“日志中是否出现成功唤醒”。

GUI 中规则输入框支持：

- 每行一条规则
- 普通关键字匹配
- 正则匹配

示例：

```text
一级唤醒成功
regex:M33_WAKEUP_AR1 success!!
```

说明：

- 普通文本按关键字匹配
- 以 `regex:` 开头的内容按正则表达式匹配

### 3.5 配置时序参数

界面中的“时序”区域包含：

- `噪声预热`：开始测试前先播放噪声多久
- `试次间隔`：相邻两次唤醒测试之间的间隔
- `成功窗口`：从播放唤醒词开始，到允许判定成功的最大时间窗口

判定说明：

- 如果匹配日志出现在成功窗口内，试次记为 `PASS`
- 如果匹配日志出现在成功窗口外，试次仍记为 `FAIL`
- 报告里会标明“窗口外命中”，方便区分“真正失败”和“超时成功”

### 3.6 配置测试场景

每个场景包含：

- 场景名称
- 噪声音频文件
- 噪声增益
- 噪声播放时长
- 唤醒词音频文件
- 唤醒词增益
- 轮数
- 是否启用

支持：

- 添加场景
- 删除场景
- 浏览噪声文件
- 浏览唤醒词文件

`噪声播放时长(ms)` 的含义：

- 填写大于 `0` 的值时，噪声会在场景开始播放后按该时长自动停止
- 噪声停止后，当前场景剩余的唤醒词试次仍会继续执行
- 留空或填 `0` 时，保持原来的默认行为，也就是噪声持续播放到整个场景结束
- 这个字段是按场景单独配置的，不影响其他场景

### 3.7 自定义次数批量设置

`自定义次数` 用于快速批量修改场景的 `轮数` 列，不需要逐行手动编辑。

当前支持三种作用范围：

- `应用到选中场景`
  - 支持一次选择一条或多条场景
  - 适合局部调参
- `应用到启用场景`
  - 只会更新 `启用` 复选框已勾选的场景
  - 适合当前回归集统一改轮数
- `应用到全部场景`
  - 会更新场景表中的所有场景

补充说明：

- 新增场景时，会继承当前 `自定义次数` 输入值
- 如果当前选中的多条场景 `轮数` 一致，输入框会自动回填这个统一值
- 如果当前选中的多条场景 `轮数` 不一致，界面会显示“试次不一致”的提示
- 混合值情况下，输入框不会被静默改成某一行的值；当前输入值会作为后续批量设置的目标值

推荐操作：

1. 先在场景表中选中目标场景，或确认哪些场景处于启用状态
2. 在 `自定义次数` 输入框中填入目标轮数
3. 点击对应的批量按钮
4. 观察状态栏提示，确认作用范围、影响数量和最终轮数

### 3.8 点击“预检”

预检会检查：

- 配置是否完整
- 音频文件是否存在且可读取
- 音频设备是否可用
- RTOS 串口或 Qualcomm ADB 日志链路是否正常
- 当前生效的日志匹配规则
- 当前全部预设参数快照

现在预检弹窗会打印一份“当前预设参数快照”，用于确认本轮真正加载的配置，包括：

- 平台
- 串口或 ADB 设备
- 音频输出设备
- 匹配规则
- `pre_noise_roll_ms`
- `trial_interval_ms`
- `success_window_ms`
- 每个场景的 `trials`
- 每个场景的 `noise_playback_duration_ms`
- 输出目录

这一步非常重要，可以避免“界面显示一套、实际运行另一套”的问题。

### 3.9 点击“开始”

点击开始后，程序会：

1. 启动日志监听
2. 播放噪声
3. 按场景和试次循环播放唤醒词
4. 捕获匹配日志
5. 生成报告

## 4. 报告输出

每轮测试默认输出到 `runs/` 目录下的一个时间戳子目录，例如：

```text
C:\Users\AORUS\Desktop\voice_wakeup_tester\runs\run_20260323_150547
```

每轮报告通常包含：

- `summary.json`
- `trial_results.csv`
- `event_log.csv`
- `run_config_snapshot.yaml`

各文件作用：

- `summary.json`：整轮统计结果
- `trial_results.csv`：每个试次的结果、时延、失败原因
- `event_log.csv`：日志原始事件与命中情况
- `run_config_snapshot.yaml`：该轮实际使用的配置快照

建议排查问题时优先看：

1. `run_config_snapshot.yaml`
2. `trial_results.csv`
3. `event_log.csv`

## 6. YAML 配置说明

可参考项目内的 [sample_config.yaml](sample_config.yaml)。

主要字段：

- `platform`：`rtos` 或 `qualcomm`
- `dut.serial_port`：RTOS 串口号
- `dut.baudrate`：串口波特率
- `dut.adb_serial`：Qualcomm 设备序列号
- `audio_devices.mouth_output`：人工嘴输出设备
- `audio_devices.noise_output`：噪声音箱输出设备
- `match_rules`：日志命中规则
- `timing.pre_noise_roll_ms`：噪声预热时间
- `timing.trial_interval_ms`：试次间隔
- `timing.success_window_ms`：成功窗口
- `scenarios`：场景列表
- `scenarios[].trials`：单个场景的播放次数
- `output_root`：报告输出根目录

## 7. 常见问题

### 7.1 预检通过，但运行结果不对

先看预检弹窗中的：

- 当前监控的日志关键词/规则
- 当前预设参数快照

重点确认：

- 匹配规则是否正确
- 成功窗口是否过短
- 场景试次次数是否正确
- 当前输出设备是否符合预期

### 7.2 实际已经唤醒，但报告仍然失败

常见原因：

- 成功日志出现在 `success_window_ms` 之外

建议：

- 查看 `event_log.csv` 是否确实有成功日志
- 查看 `trial_results.csv` 的失败原因
- 适当调大 `success_window_ms`

### 7.3 `adb logcat exited with code 255`

通常表示：

- 设备临时断连
- 设备重启
- ADB 传输异常

程序已经支持自动重试，但如果连续失败仍会报错。

建议检查：

- USB 线缆
- 设备是否在线
- `adb devices` 输出是否正常

### 7.4 蓝牙音箱重连后设备不对

蓝牙设备重连后，Windows 可能重新分配音频索引。

建议：

- 在 GUI 里点击 `刷新音频/蓝牙设备`
- 重新确认噪声音箱和人工嘴的选择
- 再执行预检

### 7.5 RTOS 没有日志

建议检查：

- 串口号是否正确
- 波特率是否正确
- 串口是否被其他工具占用

## 8. 推荐测试流程

建议每次正式测试按下面流程操作：

1. 选择平台
2. 选择音频设备
3. 配置串口或 ADB
4. 设置匹配规则
5. 设置时序参数
6. 确认场景和试次次数
7. 点击“预检”
8. 核对预检弹窗中的参数快照
9. 点击“开始”
10. 测试结束后查看报告目录
