## Context

当前停止链路分散在 GUI、引擎和音频后端三层：
- GUI 点击“停止”只会向后台线程发送 `request_stop`
- 引擎会设置 `_stop_requested`，并尽量停掉日志监听
- 循环噪声句柄支持 `stop()`，但单次唤醒词播放句柄只有 `wait()`，没有可中断接口

这会带来两个直接问题。第一，正式运行中如果正在播放唤醒词，用户点击停止后，当前音频仍可能继续播放到素材末尾。第二，试听模式本身就是同步等待单次播放结束，因此停止请求无法立刻中断试听。由于“停止”是跨模块的协同行为，这次变更需要统一梳理音频句柄契约和引擎的活动资源管理方式。

## Goals / Non-Goals

**Goals:**
- 让正式运行与试听模式中的活动音频都支持用户主动中断
- 让停止请求立即作用于当前活动音频，而不是只在后续轮询点生效
- 保证重复点击停止、正常结束后的收尾和异常路径都保持幂等
- 为停止链路建立可自动验证的回归测试

**Non-Goals:**
- 不引入暂停/恢复能力
- 不修改现有试次判定规则、成功窗口或报告结构之外的业务逻辑
- 不更换音频后端库，仍基于 `sounddevice`

## Decisions

### 1. 统一把“播放句柄”升级为可中断接口
单次播放与循环播放都应具备一致的停止能力，而不是只让循环噪声可停。为此会给播放句柄补齐 `stop()` 语义，并要求 dry-run 与真实后端都提供幂等实现。

选择这个方案，而不是仅在引擎层增加更多等待轮询，是因为“当前唤醒词仍在播”的根因来自句柄本身不可中断。只有把停止能力下沉到后端接口，GUI、引擎和试听逻辑才能复用同一套控制语义。

### 2. 引擎显式跟踪当前活动音频句柄
引擎会维护当前活动的单次播放句柄和噪声句柄，`request_stop()` 被调用时立即尝试停止这些句柄，而不是只设置 `_stop_requested` 让执行循环晚些再察觉。

选择这个方案，而不是依赖现有循环里的 `if self._stop_requested.is_set()` 检查，是因为现有检查点只覆盖“下一轮逻辑判断”之前；一旦已经进入 `playback_handle.wait()`，停止请求不会立刻作用到音频线程。

### 3. 试听模式改为走同样的可中断等待路径
试听不再只是“一次性播放然后阻塞等待”，而是复用可中断句柄与停止事件，让 GUI 的停止按钮在试听模式下也有实际效果。

选择这个方案，而不是在 GUI 层禁用试听时的停止按钮，是因为用户已经看到统一的“停止”交互，最合理的体验是无论运行还是试听，都能真正停掉当前音频。

### 4. 停止后的状态反馈保持显式且幂等
停止请求发出后，界面应明确显示“停止请求已发送”或“任务已停止”，引擎收尾阶段应允许重复调用 `stop()` 而不报错。底层音频句柄需要容忍正常结束、用户停止、重复停止这三种收尾顺序。

## Risks / Trade-offs

- [风险] `sounddevice` 回调停止存在缓冲区尾音，无法做到绝对 0 ms 静音  
  → Mitigation：把目标定义为“尽快中断并停止继续送样”，同时在测试里验证不会持续播完整段素材。

- [风险] 用户停止与自然播放结束可能并发发生，导致重复收尾  
  → Mitigation：句柄实现改为幂等停止，引擎侧只保留一份活动句柄引用，并在清理后及时置空。

- [风险] 试听与正式运行共用停止逻辑后，某一侧修复可能影响另一侧  
  → Mitigation：分别补充试听停止和正式运行停止的回归测试，确保两条路径都被覆盖。
