## Context

当前运行流程在每个场景开始时调用 `start_noise_loop()` 启动噪声循环，并在该场景所有试次完成后统一停止。这种实现对“整场景都有噪声”的回归测试足够简单，但无法表达“噪声只播放固定一段时间，后续试次继续运行”的需求。

这个变更同时跨越数据模型、YAML 配置、GUI 场景表、预检展示和执行引擎，属于一个跨模块的行为扩展。设计目标是在不改变默认行为的前提下，把噪声播放从“整场景绑定”扩展成“按场景可配置的时长窗口”。

## Goals / Non-Goals

**Goals:**

- 允许每个场景配置一个独立的噪声播放时长。
- 当配置了时长后，噪声在达到该时长后自动停止，但唤醒词试次继续按既有节奏执行。
- 让 GUI、YAML、预检快照和运行时行为对这一配置保持一致表达。
- 保持旧配置和默认行为兼容，未配置时仍然按整场景播放噪声。

**Non-Goals:**

- 不引入更复杂的噪声编排能力，例如多段噪声、按试次开关噪声、噪声淡入淡出或矩阵化时长扫描。
- 不改变 `trials`、`trial_interval_ms`、`success_window_ms` 的语义。
- 不修改日志匹配规则、统计口径和报告核心格式。
- 不新增外部依赖或后台调度线程。

## Decisions

### 1. 在 `ScenarioConfig` 中增加可选的 `noise_playback_duration_ms`

选择把噪声播放时长定义为场景级字段，而不是全局 `TimingConfig` 字段。原因是噪声文件本身就是按场景配置的，不同场景对噪声持续时长的需求也可能不同，把时长绑定到场景对象更自然。

字段采用整数毫秒值，默认 `0` 表示“按当前行为播放整个场景”。这样能保持 YAML 结构简单，旧配置无需迁移即可继续工作。

备选方案是把该字段放在 `TimingConfig` 下，作为全局统一时长，但这样无法表达“场景 A 播放 10 秒、场景 B 播放整场”的差异化需求。

### 2. 自定义时长从“噪声实际开始播放时刻”开始计时

当场景开始后，噪声一旦成功启动，就记录噪声开始播放的单调时钟时间；如果 `noise_playback_duration_ms > 0`，则在达到截止时刻后主动停止噪声句柄。

这样定义更直观，也和“噪声文件播放时长”的字面含义一致。操作者如果希望噪声覆盖前若干次唤醒试次，需要自行把 `pre_noise_roll_ms` 和试次间隔考虑进总时长。

备选方案是从第一条唤醒词开始播放时计时，但这会让“噪声预热”阶段不计入噪声时长，和用户对“噪声总共播放了多久”的直觉不一致。

### 3. 在现有主循环里检查噪声停止时机，不新增后台线程

引擎当前已经有稳定的场景循环和试次循环，因此噪声自动停止应尽量复用现有控制流：在预热等待、试次等待和试次间隔阶段检查是否达到噪声停止时间，并在首次达到时调用 `noise_handle.stop()`。

这样可以避免再引入额外线程、定时器或跨线程资源竞争，也更容易保持停止、异常和最终清理逻辑一致。

备选方案是额外起一个计时线程去异步停止噪声，但那会增加竞态、状态同步和清理复杂度，不符合当前工具“简单、可控”的实现风格。

### 4. GUI 以显式列方式展示该字段，并在预检快照中显示最终生效值

GUI 里应为场景表新增一列，例如 `噪声时长(ms)`，让操作者能直接逐场景编辑。值为 `0` 时表示“整场景”，避免再增加额外勾选框或模式下拉框。

预检快照必须逐场景显示这一字段的最终值，确保运行前就能看出该场景是“整场景噪声”还是“固定时长噪声”。

备选方案是在场景表外再增加一个附加编辑区，但会让批量场景配置的入口分裂，不利于快速对比多个场景。

## Risks / Trade-offs

- [场景表再增加一列会让 GUI 更拥挤] → 优先使用清晰命名和现有表头伸缩策略，避免引入第二套编辑面板。
- [`0` 表示整场景的语义需要用户理解] → 在列标题、工具提示和文档中明确说明 `0 = 整场景播放`。
- [噪声可能在预热阶段就提前停止] → 这是允许的显式配置结果，预检快照和文档需要说明计时起点是噪声开始播放时刻。
- [运行时停止噪声的检查点离散，可能晚几个轮询周期] → 继续使用当前主循环节奏即可，误差量级远小于现有成功窗口和试次间隔，不需要更复杂的高精度调度。

## Migration Plan

- 数据模型新增字段时提供默认值 `0`，保证旧 YAML 自动兼容。
- 配置加载时缺失该字段则回退为 `0`，保存时统一序列化。
- GUI 加载旧配置时新列默认显示 `0`。
- 如实现后需要回滚，只需忽略该字段并恢复噪声“整场景播放”逻辑；旧配置不会因此损坏。
