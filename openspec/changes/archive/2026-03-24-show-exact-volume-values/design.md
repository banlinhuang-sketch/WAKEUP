## Context

当前工具已经支持在场景表中编辑 `noise_gain_db` 和 `wakeup_gain_db`，但这些字段更偏向“原始配置输入”，不等于用户眼中的直观音量反馈。尤其在批量场景调参时，测试同学需要快速确认“当前到底生效了什么音量设置”，而不仅仅是看到一个可编辑单元格。

现有系统没有读取 Windows 系统主音量的能力，音频执行链路也完全以场景增益为准。因此这次设计聚焦于“把当前场景增益转换成更直观的具体音量显示”，而不是引入新的系统音量依赖或设备控制逻辑。

## Goals / Non-Goals

**Goals:**
- 为噪声和唤醒词增益提供更直观、可读的具体音量展示。
- 让 GUI 在单选、多选、切换场景时都能稳定反馈当前生效的音量值。
- 让预检输出也携带最终生效的音量展示信息，便于运行前核对。
- 保持现有配置模型兼容，不新增必须迁移的 YAML 字段。

**Non-Goals:**
- 不读取或控制 Windows 系统主音量。
- 不改变音频播放算法或增益计算公式。
- 不新增新的场景音量配置字段；仍以 `noise_gain_db` 和 `wakeup_gain_db` 为唯一来源。
- 不把“具体音量显示”扩展成运行中的动态电平表或波形可视化。

## Decisions

### 1. 以现有 dB 增益为唯一数据源，派生只读的“具体音量显示”
- 选择：基于现有 `noise_gain_db` / `wakeup_gain_db` 派生展示文本，例如 `-3.0 dB (0.708x)`、`0.0 dB (1.000x)`。
- 原因：当前工具的真实音量调节点就在这里，派生显示不会引入新状态，也不会造成配置和执行不一致。
- 备选方案：
  - 读取系统设备音量：更“真实”，但会引入 Windows API 依赖、设备维度差异和大量额外复杂度。
  - 只重复显示 dB：信息量不足，无法解决“具体数值不直观”的问题。

### 2. GUI 使用独立的只读“音量详情”区域，而不是直接重写表格列语义
- 选择：保留场景表中的可编辑增益列，在表格外增加只读详情展示，随当前选中场景更新。
- 原因：场景表已经承载了大量编辑字段，继续往表头塞派生字段会让横向信息过密，也不利于多选和混合值提示。
- 备选方案：
  - 在表格里新增两列派生值：实现简单，但会进一步拉宽表格。
  - 用 tooltip 展示：太隐蔽，不适合作为“确认当前音量”的主反馈。

### 3. 多选时显示“混合值”而不是虚构单一数值
- 选择：当选中多条场景且增益不一致时，详情区域显示“混合值”或逐项摘要，而不显示一个误导性的单值。
- 原因：这和现有“自定义次数”混合值处理方式一致，能避免错误判断。
- 备选方案：
  - 默认展示第一条：容易误导。
  - 禁止多选下显示详情：信息损失较大。

### 4. 预检输出补充“具体音量展示”，但底层快照仍保留原始 dB 字段
- 选择：预检面向人读，增加每个场景的噪声/唤醒词具体音量说明；`run_config_snapshot.yaml` 仍保持原始配置结构。
- 原因：预检是“核对”入口，适合放格式化文本；配置快照则更适合保持稳定、机器可读。
- 备选方案：
  - 把派生值写入 YAML 快照：会让快照出现冗余计算结果，增加维护成本。

## Risks / Trade-offs

- [“具体音量”含义被理解成系统主音量] → 在文案中明确这是“场景增益对应的有效音量显示”，不是 Windows 设备主音量。
- [多选状态下展示过于复杂] → 首版优先提供混合值提示，必要时只展示选中数量和是否一致。
- [展示格式和执行结果脱节] → 所有展示都直接基于同一套 dB 转换函数生成，避免 GUI 与引擎各算一套。
- [界面区域继续膨胀] → 采用紧凑的只读详情区，复用现有选中行同步机制，不再增加新的复杂交互。
