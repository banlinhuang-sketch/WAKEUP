## Context

当前“自定义次数”能力位于 GUI 场景编辑区，由一个 `QSpinBox` 加两个按钮组成，分别负责“应用到选中场景”和“应用到全部场景”。它已经解决了单行修改和全量统一的问题，也保留了“新增场景沿用当前次数输入值”的高效行为，但对测试同学常见的“只改一部分场景”场景支持不足。

现有实现还有两个明显约束：

- 场景表当前交互更偏向单行操作，无法把“部分批量”表达成稳定、低风险的编辑范围。
- 自定义次数输入框会根据当前选中行回填数值，当用户切到多条且次数不同的场景时，界面没有显式提示“当前是混合值”，容易误以为所有选中项已经统一。

该优化必须保持现有配置和执行模型不变，仍然以 `scenarios[].trials` 为唯一事实来源，不新增 YAML 字段，也不改变测试引擎、报告和打包逻辑。

## Goals / Non-Goals

**Goals:**

- 让测试同学可以对“选中场景”“启用场景”“全部场景”进行更高效的次数批量设置。
- 降低选中多条且次数不一致时的误操作风险，明确提示当前不是统一值。
- 在不修改配置结构的前提下提升 GUI 易用性，并保留新增场景继承当前次数输入值的现有行为。

**Non-Goals:**

- 不引入新的任务编排、矩阵运行或参数扫描能力。
- 不修改 `AppConfig`、`ScenarioConfig` 或报告文件结构。
- 不新增复杂的算术编辑能力，例如“递增 1”“按比例放大”或表达式输入。
- 不改变 headless/CLI 对 `scenarios[].trials` 的读取语义。

## Decisions

### 1. 继续以场景表中的 `trials` 列作为唯一数据源

选择保留当前模型，不增加“全局默认次数”或“批量设置历史”字段。这样可以确保 GUI、YAML、预检快照、执行引擎和报告继续围绕同一个值工作，避免引入新的优先级问题。

备选方案是新增一个独立的默认次数配置并在运行前覆盖场景值，但这会让界面显示和实际执行之间再次产生错位风险，与当前工具“预检展示实际生效配置”的方向相冲突。

### 2. 批量编辑范围固定为三个显式作用域

自定义次数优化后的批量目标固定为：

- 选中场景
- 启用场景
- 全部场景

这样能覆盖测试同学最常见的三类操作：局部调参、只改当前启用集、统一全量回写，同时避免引入过度通用的筛选器或规则表达式。

备选方案是只保留“选中 / 全部”，但这无法解决“只改当前启用集”的高频需求；另一种备选是做任意条件过滤，但首轮会让界面和实现复杂度明显上升。

### 3. 多选与混合值反馈优先于新增复杂控件

场景表需要支持多行选择，并增加一个轻量级的混合值提示机制：

- 若选中范围内 `trials` 全部一致，则自定义次数输入框回填该值。
- 若选中范围内 `trials` 不一致，则不再假装存在统一值，而是通过状态文本或辅助提示明确显示“选中场景次数不一致”。

这里选择“提示混合状态 + 保留当前输入框作为待应用值”，而不是引入三态输入框或额外弹窗。这样对现有界面侵入更小，学习成本也更低。

### 4. 反馈信息必须包含作用范围和受影响数量

每次批量应用完成后，状态栏或提示文本都应明确说明：

- 本次应用到了哪一类范围
- 一共修改了多少条场景
- 设置成了多少次

例如：“已将 6 条启用场景的试次设置为 20 次”。

备选方案是继续只显示“已设置为 X 次”，但当批量范围变多后，这种反馈不足以帮助用户确认本次操作是否命中了预期对象。

## Risks / Trade-offs

- [多选交互改变现有表格行为] → 通过集中封装“作用域取行逻辑”，减少代码里对 `currentRow()` 的隐式依赖。
- [混合值提示不够显眼] → 在输入框附近增加可见文本提示，并在状态栏同步说明，避免只靠一次性消息。
- [按钮数量增加导致界面更拥挤] → 维持现有紧凑布局，只新增必要的一个作用域按钮或把作用域选择收敛为更清晰的操作组。
- [没有新增配置字段会限制后续高级能力] → 首轮刻意保持简单，后续如要做任务集/矩阵能力，再在更高层抽象中扩展。
